# Group-Bot Coverage + Delta (missed) Report

## GROUP BOT — what it CAN do
- CAPTURE FROM CHATTER (the core job). app/api/group/ingest/route.ts stores EVERY group message (messages row, sender_type='group', account=group, handled_by='group-bot') and emits whatsapp.group_in so it lands on the sender's profile timeline, then wakes the SAME brain (runSasa surface='group', One-brain law) on substantive messages. substantive() (lines 51-57) gates the brain on cost: fires if addressed 'sasa', ends in '?', length >= 25, OR has a quoted reply / @mention (line 260). buildGroupSystem captureLaw (sasa.ts 334-341) forces tools-FIRST: an assignment -> create_task(assignee_name, due_on); a 'done' -> complete_task; an undo/'not done' -> reopen_task; a beneficiary/stock -> add_beneficiary/add_inventory_item; a durable org fact -> remember_fact. The honesty guard (claimsCompletionWithoutSuccess, sasa.ts 95-111) neutralizes any 'noted/tracked/done' claim unless the matching tool returned ok=true THIS turn.
- ANSWER TEAM READS (team-tier). list_tasks, team_detail (role/phone, pay STRIPPED for team), lookup_contact (a colleague's number), list_campaigns (names only, money stripped), query_calendar (payroll shows as dateless '<category> day', no amount), check_conflicts (holiday/load only). Reads are free, no confirmation. The prompt tells it to look up and answer in ONE short warm sentence.
- ONE-LINE CONFIRMS / @MENTIONS. After a successful capture it confirms in exactly one warm line that @mentions the person (e.g. 'Noted @Cynthia, tracked: stall map, due Thu.'), never listing tool names, never revealing it is an AI.
- STAY SILENT (the default). buildGroupSystem instructs reply with exactly NO_REPLY unless addressed, a direct answerable question, or a brief confirm; 'when unsure, prefer NO_REPLY'. The reply gate in runSasa (sasa.ts 459) returns empty reply on /^NO_REPLY$/ AFTER tools ran, so capture still happens while the group hears nothing. Even unactionable items (links, images, forwards) are captured by staying silent, never apologized for.
- ESCALATE PRIVATELY (FLAG_NUR). High-stakes AND low-confidence -> model returns 'FLAG_NUR: <reason>' (sasa.ts 462 returns it raw, skipping humanize/verify). ingest (301-314) strips the prefix and WhatsApps the reason to WHATSAPP_OPERATORS on the 727, never to the group, with 1-hour dedup on events type group.flagged_nur, and fires EVEN in listen-only.
- INGEST MEDIA / VOICE / LINKS without speaking. Voice notes transcribed (transcribeAudio) and treated as text; images+docs uploaded to assets bucket and pushed through the shared createBatch ingest pipeline (auto-file); shared-link WhatsApp preview folded into the text + a whatsapp.group_link_in event; a positive reaction (check/thumbs) on a message replays it through complete_task. All silent, all best-effort with whatsapp.group_media_failed events on error (no swallowed failures).
- CASES INTAKE (Rescue & Rehab etc.). In GROUP_CASE_LIST groups (default 'nisria • rescue & rehab') the brain runs casesIntake=true: add_beneficiary lands an under_review CASE on /cases (status inactive, case_channel='group:<name>', consent_public false), never an active beneficiary; case photos route to storeCaseGroupPhoto / attachPendingCasePhotos; history is suppressed ([]) so the same child is not re-logged each turn. Always silent in-group.
- QUEUE PORTAL->GROUP POSTS (one-way). post_to_group (smart-tools 607-614) and the digest insert kind='group.send' jobs; /api/group/outbox GET lets the Railway userbot claim up to 20 queued jobs (marks 'sending', self-heals 'sending' older than 5min back to 'queued'), POST acks done or re-queues up to 5 attempts. The portal never touches WhatsApp directly; the userbot is the sole sender.
- DAILY DIGEST. /api/group/digest (Vercel cron / agent secret / group secret) batches each group's due-today + overdue tasks into ONE @mentioned morning post per group, idempotent per day (digest_date in payload), 'today' computed in Nairobi UTC+3.
- BOT LIVENESS + LINK. /api/group/link relays the userbot's short-lived QR to the Groups page (session-cookie gated GET) and latches one urgent 727 alert to operators on banned/logged_out; outbox GET writes a ~heartbeat (bot_status group_poll) so link GET reports 'connected' from a fresh poll even if a stale flag says 'waiting'.

## GROUP BOT — what it CANNOT do (why)
- **See or speak ANY money, donor, donation, grant, salary, or beneficiary detail in a group** — Hard wall by THREE layers. (1) Tier: inGroup forces role='team' regardless of who posts (sasa.ts 377-378), and tools are filtered to TEAM_TOOL_NAMES (sasa.ts 37-38, 397) which excludes every finance/donor read (query_donations, finance_summary, lookup_donor, list_grants) and message/draft sends. (2) Grounding strip: team memories with finance vocabulary or a material money figure are removed before the prompt (carriesMoney / FINANCE_GROUNDING / MONEY_FIGURE, sasa.ts 46-56, 392). (3) Tool-level: group_activity/member_activity return {error:'not available here'} for tier team, list_campaigns/team_detail strip money/pay, find_beneficiary/add_beneficiary read path is blocked for team (smart-tools 267,281,305,321,344,348,364). The prompt restates beneficiaries are confidential child-safeguarding data: never name a child, story, location, or contact in a group.
- **Reply inside a group at all, right now** — LISTEN_ONLY is on by default: GROUP_LISTEN_ONLY defaults true unless explicitly 'false' (ingest line 28). While true the brain still runs and captures, but ingest returns reply:'' and logs no outbound (lines 318-320). Per-group GROUP_MUTE_LIST and all case groups are also forced silent. So today the bot is a silent capturer + private escalator; in-group speech requires flipping the env flag.
- **Send a 1:1 WhatsApp, an email, post to a group on its own initiative, or move money** — message_person, post_to_group, draft_email, draft_thank_you, record_payment are NOT in TEAM_TOOL_NAMES, so the group brain cannot call them. captureLaw ends 'Never claim you sent an email or moved money.' Outbound to a group only happens when an OPERATOR (Nur/Taona) on the 727 explicitly drives post_to_group, queued one-way via the outbox.
- **group_activity / member_activity (read what the groups are doing)** — Those two tools are admin-only by design (return error for tier team). They are the OPERATOR's window into group activity from the 727/console, not something the group bot exposes to teammates. The group bot itself runs at team tier so it cannot call them.
- **Create recurring tasks/reminders/events from a group** — Platform holds ONE date per task/event; the prompt instructs setting the next single date, stating recurring is unsupported, and not looping. No recurrence engine exists.
- **Reliably attribute a message to the right team_member** — Identity is self-healing and best-effort: learnMemberPhone (ingest 75-86) only binds a phone to a member when exactly ONE active, phone-less name matches the sender's first name. Until a member's phone is learned, speakerPhone-based 'tick the speaker's own task' and member_activity correlation can miss.

## GROUP BOT — permissions
Three concentric gates. (1) TRANSPORT AUTH: every group route checks x-group-secret === GROUP_BOT_SECRET (ingest, outbox, link POST); digest also accepts CRON_SECRET bearer / AGENT_TICK_SECRET; link GET additionally accepts the nisria_session cookie because the QR is a linking secret. Service-role only, never client-exposed (returns 401 otherwise, default secret '\\0' so an unset env never matches). (2) TIER (the PII wall in a group): inGroup => role='team' unconditionally (a group is team-tier regardless of who posts, even Nur/owner), rank forced to null into runSmartTool. Team tier = TEAM_TOOL_NAMES only (15 tools: list_tasks, create_task, complete_task, reopen_task, add_beneficiary, add_inventory_item, team_detail, lookup_contact, list_campaigns, remember_fact, query_calendar, check_conflicts, create_event/move_event/delete_event). Calendar tools are deliberately team-safe (payroll dateless+amountless, events never touch finance rows). Grounding stripped of money. Tool bodies independently re-enforce the wall (error/strip for tier team) so a prompt jailbreak still cannot read finance/donor/beneficiary data. viewerIsOwner=false for team, so the owner's 727 line is excluded from any message-log read. (3) NO_REPLY GATE (the speech wall): the model must emit NO_REPLY in all but three situations; tools run BEFORE the gate so capture is decoupled from speaking; the gate returns empty reply on NO_REPLY and routes FLAG_NUR privately to operators; on top of that LISTEN_ONLY / MUTE_LIST / case-group all hard-suppress in-group sends. Outbound to a group is one-way and operator-initiated only (post_to_group/digest -> jobs -> userbot via outbox).

## GROUP BOT — gaps (14)
- [high|platform] Group reply path bypasses the action gateway/intents. ingest queues group.send jobs and writes messages directly rather than going through lib/gateway.ts createIntent, so group sends are not subject to autonomy lanes, action_intents logging, or the duplicate guard the rest of the platform uses.
  - *Fix:* Route post_to_group / digest sends through createIntent({connector:'whatsapp-group', action:'group.send', idempotency_key,...}) so group delivery shares the platform's audited, idempotent, lane-governed path.
- [medium|platform] Identity binding is fragile: complete_task/reopen_task and member attribution depend on learnMemberPhone, which only fires on a single unambiguous phone-less first-name match. A common first name, or a member whose phone is already wrongly bound, leaves the speaker unresolved and the 'tick your own task' shortcut broken.
  - *Fix:* Add an admin surface on /groups (or /team) to manually confirm/correct phone<->member bindings, and surface a 'who am I matching' diagnostic. Optionally seed phones from the roster proactively rather than only learning from live traffic.
- [medium|platform] FLAG_NUR escalation has no audit/visibility surface for the operator beyond the raw 727 text + an events row; if WHATSAPP_OPERATORS is empty or sendText fails (swallowed by try/catch, ingest 310), the flag is silently lost.
  - *Fix:* Persist every FLAG_NUR as a Needs-You / approvals item (not just an event) so it is durable and visible in the portal even if the WhatsApp push fails; surface failed-notify count in the bot_status / Groups page.
- [medium|platform] substantive() can miss real intent: a short report under 25 chars with no '?', no 'sasa', no quote/@mention (e.g. 'paid the rent', 'kid intake done') never wakes the brain, so the capture silently doesn't happen and there is no trace that a message was skipped.
  - *Fix:* Add a lightweight verb/keyword classifier (done|paid|finished|added|intake|received) to the substantive gate, or periodically batch-replay stored-but-unwoken group messages so nothing intentful is permanently dropped.
- [medium|runtime] Daily digest @mentions do not actually notify anyone. The digest text contains plain '@FirstName' but the bot's send path posts { text } with no WhatsApp `mentions` jid array, so assignees are not pinged, only visually tagged as inert text.
  - *Fix:* Thread the assignee jids through the outbox payload (e.g. payload.mentions = [jid]) and have humanSend pass { text, mentions } to sock.sendMessage. Requires resolving member phone -> jid (the bot already has nameToJid for groups; add a phone->jid map or build mention jids as `${digits}@s.whatsapp.net`).
- [medium|runtime] Inbound message dedup lives only on the platform (messages.external_id). If /api/group/ingest is slow/times out (maxDuration 300s) the bot has no local idempotency and no retry; a transient ingest failure drops the message silently (returns { reply: '' }). The bot also has no inbound queue, so a burst during a platform outage is lost entirely.
  - *Fix:* Add a small local dedup/seen-set keyed by message_id and a bounded retry/backoff (or a disk-backed pending queue flushed on reconnect) so platform hiccups do not lose group messages. At minimum, log dropped messages with their ids for replay.
- [medium|runtime] Silent send failures on the inline reply path. humanSend wraps sock.sendMessage in try/finally but the inline-reply call site (messages.upsert) does not check success: if the send throws it is caught by the outer per-message try/catch and logged generically, but the brain already considered it 'replied' and the platform logged an outbound message row in ingest. Outbox path acks failures (good), but the inline reply path has no ack/requeue, so a failed autonomous reply is lost and the thread/counts show a phantom outbound.
  - *Fix:* Have humanSend return a boolean; only log 'replied' on success, and on the inline path route failed sends through the same requeue mechanism as the outbox (or emit a failure event so the platform can reconcile the phantom outbound row). Note: currently LISTEN_ONLY defaults true so inline replies are effectively off, which masks this, but it activates the moment GROUP_LISTEN_ONLY=false.
- [medium|runtime] GROUP_BOT_SECRET is a single static shared secret with no rotation story and is reused across four endpoints including the cron digest. A leak (and per MEMORY there has been prior secret-leak hygiene risk in this project) gives an attacker full ingest/outbox/link/digest control, including the ability to inject group.send jobs or post a fake QR/banned status to alarm operators.
  - *Fix:* Rotate the secret on both Railway and Vercel together (it must match exactly); add timing-safe comparison server-side; consider per-purpose secrets or signed requests (HMAC over body + timestamp) so a replayed body can't be re-posted, and audit that no real .env with the secret is committed anywhere in the monorepo.
- [low|platform] In-group replies are globally disabled (GROUP_LISTEN_ONLY defaults true). The bot's 'one-line confirm' and 'answer team reads' capabilities are dark in production until the flag is flipped.
  - *Fix:* Once trust is established, set GROUP_LISTEN_ONLY=false in Vercel env (verify no trailing \n/quote per the known Vercel bug). The dequote/heartbeat machinery is already built to support live replies; no code change needed.
- [low|platform] No idempotency on group.send beyond the digest's per-day key. post_to_group inserts a fresh job every call, so a retried operator action or a double tool-call could double-post into a group. (Contrast: lib/CLAUDE.md mandates idempotency on every external action, and outbox notes 'inbound dedupe at send time' but the queue side has none for post_to_group.)
  - *Fix:* Add an idempotency key to post_to_group jobs derived from (group, text, time-window) and have the insert short-circuit on collision, matching the gateway/intents pattern the doctrine requires.
- [low|runtime] Outbox group-name resolution is fuzzy and can mis-target. resolveJid falls back to bidirectional substring matching (k.includes(n) || n.includes(k)); two similarly named groups (e.g. 'Nisria' vs 'Nisria Rescue') could resolve a portal send into the wrong group. There is no jid pinning back to the portal.
  - *Fix:* Prefer exact match, and when only a substring match exists require it to be unambiguous (exactly one candidate) else ack as error 'ambiguous group'. Better: have the portal store and target by jid once primed, not by free-text name.
- [low|runtime] QR re-link has a stale-secret window and depends on a human being present. The QR only lives ~30s; the bot re-pushes on every refresh, but if the portal /api/group/link POST fails (network) the relayed QR goes stale and Nur sees an expired code. There is no auto-rotation surfaced as 'expired' to the portal, and no fallback delivery (the README's email/Desktop preference for human steps is not used here).
  - *Fix:* Stamp each relayed QR with a short TTL the portal can show as 'refreshing…' and auto-expire; consider falling back to the PAIR_NUMBER pairing-code path (already implemented) as the primary re-link method since codes are more forgiving than a 30s photo.
- [low|runtime] Multi-group at scale: nameToJid is primed once on connect via groupFetchAllParticipating; a group the bot is ADDED to after connect (or renamed) is only learned lazily on its first inbound message (groupName via groupMetadata). An outbox send to a freshly-added group before any inbound is seen will fail 'unknown group'.
  - *Fix:* Re-prime the name->jid map on a 'groups.upsert'/'groups.update' event and periodically, or fall back to a live groupMetadata lookup inside resolveJid when the name is unknown.
- [low|runtime] Restart amnesia for outbox in-flight + reconnect double-binding risk. On reconnect the code clears and re-sets pollTimer (good), but if start() is re-entered on a benign drop while a prior socket's async handlers are still resolving, two message handlers could briefly coexist (the reconnect creates a NEW sock with NEW ev listeners; the old sock is not explicitly removed/ended). Could double-process a message during the overlap window.
  - *Fix:* Explicitly sock.ev.removeAllListeners() / sock.end() before scheduling the reconnect start(), or guard start() with a single-flight lock so only one live socket exists at a time.
